# Reflexion Agent (RAG + Tool Execution + Self-Improvement)

An AI agent system that performs **multi-step reasoning**, **retrieval-augmented generation (RAG)**, and **tool execution**, enhanced with a **Reflexion loop** to iteratively improve its outputs.

---

## What This Project Does

This project builds an AI agent that can **reason through problems, retrieve relevant information, execute tools when needed, and then improve its own answers through a Reflexion loop**. Instead of generating a one-shot response, the system follows a multi-step workflow: it analyzes the user query, gathers supporting context via retrieval (RAG), performs actions through a controlled tool execution layer, and produces an initial answer. It then critiques that answer, identifies weaknesses, and refines it to produce a more accurate and grounded final result. Overall, the project demonstrates how to design a **modular, production-style AI agent system** that combines reasoning, retrieval, and self-improvement to deliver higher-quality responses.

---

## Key Features

- **Multi-step reasoning agent**
  - Breaks down a problem into steps
  - Doesn’t just generate one-shot answers
  - Shows agent-style thinking, not simple prompting

- **Retrieval-Augmented Generation (RAG)**
  - Retrieves relevant context from a vector store
  - Uses that context to ground responses
  - Reduces hallucination and improves accuracy

- **Tool calling / execution layer**
  - Agent can call external tools or functions
  - Separates:
    - reasoning (LLM)
    - execution (system)
  - This is core to real-world AI agents

- **Reflexion loop (most important feature)**
  - Generates an initial answer
  - Critiques its own output
  - Refines and improves the answer
  - Enables self-improving behavior without retraining

- **Modular architecture**
  - `chains.py` → reasoning logic
  - `tool_executor.py` → execution
  - `schemas.py` → structured data
  - `reflexion_agent` → improvement loop
  - Easy to extend and debug

- **Structured outputs (schemas)**
  - Enforces consistent input/output formats
  - Makes tool calling and reasoning reliable

- **Iterative decision flow**
  - Decide → retrieve → act → evaluate → improve
  - Not linear — behaves like a real system

- **Separation of concerns**
  - LLM decides what to do
  - System enforces how it’s done
  - This is production-level design thinking

## LangGraph Workflow

The LangGraph workflow is a draft → search tools → revise → repeat loop.

```text
START
  ↓
draft
  ↓
execute_tools
  ↓
revise
  ↓
event_loop
  ├── execute_tools  (continue improving)
  └── END            (stop after max iterations)
```

- draft node

  Creates the first answer using `first_responder`. The model is forced to output a structured `AnswerQuestion` tool call containing:   answer, reflection, and search queries.

- execute_tools node

  Runs Tavily Search using the search queries generated by the model. This gives the agent external information to improve the answer.

- revise node

  Uses `revisor` to revise the original answer based on tool results. The revised output must include citations/references through      the `ReviseAnswer` schema.

- event_loop conditional edge

  Counts how many ToolMessages exist in the graph state. If the number of tool visits is greater than `MAX_ITERATIONS = 2`, the         workflow ends; otherwise, it goes back to `execute_tools`.

## Set up

1. Clone the repository

```text
git clone https://github.com/nut-hazel-ll/reflexion-agent.git
cd reflexion-agent
```

2. Install dependencies

Using Pipenv:

```text
pip install pipenv
pipenv install
pipenv shell
```

3. Create a `.env` file

Create a `.env` file in the project root:

```text
OPENAI_API_KEY=your_openai_api_key
TAVILY_API_KEY=your_tavily_api_key
LANGCHAIN_API_KEY=your_langsmith_api_key_optional
LANGCHAIN_TRACING_V2=true
LANGCHAIN_PROJECT=reflexion-agent
```

Do not commit `.env`.

4. Run the project

```text
python main.py
```

## Design Decisions

### Draft first, then improve

The agent does not try to produce the perfect answer immediately. It first creates an initial answer, then reflects on weaknesses and improves it.

### Use structured reasoning

```text
answer
reflection
search_queries
```

This makes the agent’s reasoning more controllable and easier to debug.

### Search when more evidence is needed

If the draft identifies missing information, the agent generates search queries and uses Tavily Search to collect external evidence.

### Separate reasoning from tool execution

The LLM decides what information it needs, but the tool executor actually performs the search. This keeps execution controlled and safer.

### Revise based on evidence

After receiving search results, the agent updates the answer and adds stronger support/citations instead of blindly trusting the first response.

### Stop after a fixed number of iterations

The workflow uses a max-iteration limit, so the agent does not loop forever. This balances answer quality, latency, and cost.

### Use LangGraph to control the loop

LangGraph manages the flow:

```text
draft → execute_tools → revise → check whether to continue → end
```
